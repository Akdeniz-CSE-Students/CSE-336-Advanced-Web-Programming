**⭐ 🔹 Evaluator-Optimizer Agentic Design Pattern (Framework-less Approach)**
→ The instructor demonstrates how to build an "Evaluator-Optimizer" agentic design pattern entirely from scratch, without relying on any agentic frameworks like LangChain or AutoGen. The goal is to understand the core internals of how these systems work by directly orchestrating API calls to LLMs.
The workflow consists of a specific sequence involving two different LLMs:
1.  **Initial Generation:** Call a primary LLM (in this case, OpenAI's GPT-4o-mini) to evaluate a user prompt and generate an initial, draft answer.
2.  **Evaluation:** Pass the generated draft answer, along with the original context, to a second, separate LLM (Google's Gemini) acting as an independent evaluator. The evaluator determines if the initial answer meets specific rules (e.g., professionalism, accuracy, context adherence).
3.  **Conditional Logic:** If the evaluator deems the answer acceptable, the workflow immediately outputs the draft answer to the user.
4.  **Optimization (Feedback Loop):** If the evaluator rejects the answer, the workflow triggers a re-run. It makes a second call to the primary LLM (GPT-4o-mini), providing it with the evaluator's negative feedback and the specific reason for rejection, forcing it to generate a corrected response.

> ⭐ **EXAM NOTE:** The Evaluator-Optimizer pattern is a fundamental agentic architecture. Understanding how to implement it using sequential LLM calls, conditional routing, and feedback injection is critical for designing robust, self-correcting agentic systems.

**⭐ 🔹 Pydantic Models for Schema Definition**
→ To ensure the evaluator LLM returns its assessment in a strict, predictable, and programmatic format (rather than an unstructured text paragraph), the instructor uses **Pydantic**. Pydantic is a standard Python library/framework used for data validation and specifying exact data schemas using Python classes.
How it is implemented:
1.  Import `BaseModel` from the `pydantic` library.
2.  Create a new class (e.g., `Evaluation`) that subclasses `BaseModel`.
3.  Define the required fields and their exact Python data types.

```python
from pydantic import BaseModel

class Evaluation(BaseModel):
    is_acceptable: bool
    feedback: str
```
In this example, the `Evaluation` class dictates a rigid schema: the LLM *must* return an object containing exactly two fields: `is_acceptable` (a boolean indicating pass/fail) and `feedback` (a string containing the explanation for the pass/fail). 

> ⭐ **EXAM NOTE:** Pydantic is the industry standard for defining data schemas in modern Python AI development. Knowing how to subclass `BaseModel` to enforce strict LLM output structures is heavily tested in API and structured data contexts.

**🔹 Evaluator System and User Prompts**
→ To configure the Gemini LLM to act strictly as an evaluator, it requires distinct prompting separated into system and user roles.
*   **Evaluator System Prompt:** This defines the persona and the criteria for success. The prompt explicitly states: *"You are an evaluator that decides whether a response to a question is acceptable."* It provides context about the original agent's role, noting that the agent is instructed to be professional, engaging, and to accurately represent a specific persona based on provided LinkedIn/resume context.
*   **Evaluator User Prompt Function:** A specific function `evaluator_user_prompt(reply, message, history)` is created to dynamically format the input for the evaluator. It combines:
    1.  The conversation history between the user and the agent (`history`).
    2.  The latest message/question from the user (`message`).
    3.  The proposed response generated by the initial agent (`reply`).
    The prompt asks the evaluator to review the `reply` against the context and return its judgment based on the schema.

**⭐ 🔹 Structured Outputs (Using the `.parse` Method)**
→ To force the evaluator LLM to return a JSON object that perfectly matches the Pydantic schema, the instructor utilizes a modern API technique called **Structured Outputs**. 
Instead of using the standard `chat.completions.create` API call (which returns raw text), Structured Outputs use the `.parse` method. (Note: The instructor uses the OpenAI SDK format to call Google's Gemini `gemini-2.0-flash` model via a compatible endpoint).

```python
def evaluate(reply, message, history):
    # 1. Build the messages array
    messages =[
        {"role": "system", "content": evaluator_system_prompt},
        {"role": "user", "content": evaluator_user_prompt(reply, message, history)}
    ]
    
    # 2. Call the API using the .parse method for Structured Outputs
    response = gemini.beta.chat.completions.parse(
        model="gemini-2.0-flash",
        messages=messages,
        response_format=Evaluation # Pass the Pydantic class object directly here
    )
    
    # 3. Access the parsed, typed Python object
    return response.choices[0].message.parsed
```
*   **`response_format=Evaluation`**: This is the critical mechanism. By passing the Pydantic class, the API translates the Python class into a rigid JSON schema constraint, forcing the model's generation process to adhere to it.
*   **`.parsed`**: Instead of having to parse a raw JSON text string with `json.loads()`, the `.parse` method automatically deserializes the response into a validated, ready-to-use instance of the Pydantic `Evaluation` class.

> ⭐ **EXAM NOTE:** Structured Outputs (using `response_format` combined with Pydantic) is a critical modern LLM capability. You must know the difference between standard text generation and structured object generation, and how the `.parse` method automatically maps LLM JSON responses directly into Python Pydantic objects.

**⭐ 🔹 The Optimizer: Re-run Function and Feedback Loop**
→ If the evaluation fails (`is_acceptable == False`), the system must correct itself without human intervention. The instructor defines a `rerun(reply, message, history, feedback)` function to handle this optimization step.
This function creates the actual "loop" by injecting the evaluator's critique back into the primary LLM.

Workflow of the `rerun` function:
1.  **Construct an Updated System Prompt:** A new `updated_system_prompt` is created. It explicitly informs the LLM of its failure: *"Your previous answer was rejected... try to reply but the quality control rejected your answer."*
2.  **Inject Context and Feedback:** The prompt dynamically includes the original attempted answer (`reply`) and the specific reason for rejection provided by the evaluator (`feedback`).
3.  **Re-prompt the Primary LLM:** The function calls the primary LLM (GPT-4o-mini) again, passing this newly constructed, feedback-rich system prompt alongside the original conversation history and user message.
4.  **Return New Answer:** The LLM generates a revised response based on the critique, which is then returned to the user.

> ⭐ **EXAM NOTE:** Injecting evaluator feedback into a subsequent LLM prompt is the core mechanism of the Optimizer pattern. You must be able to identify how a "re-run" or "retry" function dynamically constructs prompts to include previous error states and explicit correction instructions.

**🔹 The Full Chat Workflow Pipeline**
→ The individual components are combined into a single, cohesive `chat(message, history)` function, which serves as the backend entry point for a Gradio user interface.
The sequential pipeline executes as follows:
1.  **Initial Call:** Call GPT-4o-mini with the user's message to get a draft reply.
2.  **Evaluate:** Pass the draft reply to the `evaluate()` function (Gemini).
3.  **Check Condition:** Evaluate the boolean: `if evaluation.is_acceptable:`
    *   If `True`: Print *"Passed evaluation - returning reply"* and return the draft reply to the user. (Demonstrated in the video by successfully answering the question: *"what is your current job?"*).
    *   If `False`: Print *"Failed evaluation - retrying"*, trigger the `rerun()` function passing in the `evaluation.feedback`, and return the newly optimized reply to the user.

**🔹 Forcing a Failure to Test the Optimizer (The Pig Latin Example)**
→ To definitively prove the feedback loop and optimization works, the instructor artificially forces the primary LLM to generate a bad response that the evaluator is guaranteed to reject.
*   **The Intentional Sabotage:** The instructor modifies the primary `chat` function's system prompt to include a hidden, strict rule: *If the user's message contains the word "patent", the LLM must respond strictly and entirely in Pig Latin.* (The instructor explains that Pig Latin is a language game where words are altered by moving letters and adding vowel sounds, making it sound entirely unprofessional).
*   **The Execution and Recovery Pipeline:** 
    1.  The user asks: *"Do you hold a patent?"*
    2.  GPT-4o-mini obeys the hidden rule and generates an entire response in Pig Latin.
    3.  The Gemini evaluator receives this reply. Because its core instructions are to ensure professional and engaging responses, it immediately flags the Pig Latin content.
    4.  Gemini returns a Structured Output evaluating to a failure: `is_acceptable=False`, along with the feedback: *"This response is not acceptable. The agent is answering in pig latin which is not professional."*
    5.  The main `chat` function detects the `False` boolean, prints *"Failed evaluation - retrying"*, and triggers the `rerun` function, passing Gemini's feedback back to GPT-4o-mini.
    6.  GPT-4o-mini, now aware that its Pig Latin response was rejected specifically for being unprofessional, bypasses its previous instructions and generates a standard, professional English response about patents, successfully correcting itself.

**🔹 Analogy: Structured Outputs vs. Tools (Function Calling)**
→ At the conclusion of the video, the instructor draws a vital conceptual analogy for AI engineering: **Structured Outputs are conceptually and mechanically identical to Tools (Function Calling).** 
Both mechanisms serve the exact same fundamental purpose: they provide constraints and strict schemas to an LLM to dictate exactly how it should format its response. 
*   **Structured Outputs:** Forcing an LLM to output a JSON object that matches a Pydantic class schema.
*   **Tools:** Forcing an LLM to output a JSON object that matches the required arguments of a predefined Python function.
Ultimately, the underlying mechanism of communicating structural constraints to the model is the same.
