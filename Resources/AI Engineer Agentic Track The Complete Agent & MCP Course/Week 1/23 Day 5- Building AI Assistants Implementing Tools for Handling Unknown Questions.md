**🔹 Handling Tool Calls (Helper Function)**
→ Visible at the very beginning of the video, this helper function is responsible for executing the tools the LLM requests. It takes the list of `tool_calls` generated by the model, iterates through them, extracts the function name and arguments, and dynamically fetches the corresponding Python function from the script's global namespace using `globals().get(func_name)`. It then executes the function with the parsed arguments and appends the result into a specific dictionary format (with `"role": "tool"`) required by the LLM to understand the tool's output.
```python
def handle_tool_calls(tool_calls):
    results =[]
    for tool_call in tool_calls:
        func_name = tool_call.function.name
        arguments = json.loads(tool_call.function.arguments)
        print(f"Tool called: {func_name}", flush=True)
        tool = globals().get(func_name)
        result = tool(**arguments) if tool else ""
        results.append({"role": "tool", "content": json.dumps(result), "tool_call_id": tool_call.id})
    return results
```

**⭐ 🔹 The Principle of Repetition in Prompting**
→ The instructor highlights an important nuance: theoretically, putting tool directives into the system prompt isn't strictly necessary. This is because the JSON schema used to define the tool to the LLM already provides context, a description of the tool, and explains when it should be used.
However, the instructor emphasizes that **it never hurts to be repetitive in prompting.** 
Explaining the rules several times over (once in the JSON tool description, and again in the system prompt) significantly increases the probability that the model will perform exactly the way you want it to. It strongly biases the model to output tokens that are consistent with your objective.

> ⭐ **EXAM NOTE:** The concept that intentional redundancy (repeating instructions in both tool schemas and system prompts) increases the probability of desired model behavior is a fundamental best practice in robust prompt engineering for agents.

**⭐ 🔹 The Illusion of True Autonomy vs. Statistical Next-Token Prediction**
→ The instructor provides a profound philosophical and technical sidebar about how LLMs actually operate when "making decisions." 

*   **The Illusion (Anthropomorphism):** We frequently use language like "the model calls a tool" or "the agent decides to use a tool." This makes it sound like the AI possesses true autonomy, consciousness, or independent decision-making capabilities.
*   **The Reality (Next-Token Prediction):** An LLM is purely a statistical engine. All it is doing is taking an input sequence of tokens and generating the most likely, predictive next tokens to follow that sequence. 

Because the input prompt contains descriptions of available tools and explicit directives to use them, the statistical "most likely next text" naturally becomes the structured tokens that invoke the tool. The model isn't "thinking" about using a tool; it is just outputting the statistically correct text based on the provided context. The instructor admits that it "hurts the mind" to realize that simple next-token prediction can achieve complex, seemingly autonomous actions like tool calling, but this is the fundamental mechanical reality of LLMs. (The instructor also notes they have YouTube videos diving deeper into how next-token prediction achieves this).

> ⭐ **EXAM NOTE:** Understanding this distinction is absolutely mandatory. Agentic behavior is mechanically just statistical next-token prediction based heavily on the input context (prompts + tool schemas). Recognizing this prevents engineers from treating models like conscious entities and focuses debugging on context and token probabilities.

**🔹 The Agentic Chat Loop (Preview)**
→ The video ends by arriving at the `chat(message, history)` function, which the instructor notes will be the focus of the next lesson. This block of code shows the foundational `while not done:` loop pattern that drives agentic systems. It sends the prompt and tools to the OpenAI API, checks if the model's `finish_reason` is `"tool_calls"`, executes those tools, appends the results to the message history, and loops again until the model returns a final text response (`done = True`).
```python
def chat(message, history):
    messages =[{"role": "system", "content": system_prompt}] + history + [{"role": "user", "content": message}]
    done = False
    while not done:
        response = openai.chat.completions.create(model="gpt-4o-mini", messages=messages, tools=tools)
        if response.choices[0].finish_reason == "tool_calls":
            messages.append(response.choices[0].message)
            results = handle_tool_calls(response.choices[0].message.tool_calls)
            messages.extend(results)
        else:
            done = True
    return response.choices[0].message.content
```
