**🔹 Initializing a CrewAI Project via CLI**
→ To create a new CrewAI project from scratch, you utilize the command line interface. 
- Open the terminal (using `Ctrl + \`` on a Mac or navigating via the View menu).
- Change into your target directory (e.g., `cd 3_crew`).
- Run the creation command: `crewai create crew [project_name]`. 
  - *Example used in video:* `crewai create crew debate`
- **Scaffolding Prompts:** The CLI will ask for your preferred starting LLM provider (e.g., OpenAI) and the specific model (e.g., `gpt-4o-mini` for cost efficiency). 
- **API Key:** It will prompt for your API key. If you already have a `.env` file configured in your broader workspace managing your keys, you can simply press `Enter` to skip this step and prevent it from generating redundant `.env` files.

**🔹 IDE/Cursor Directory Exploration Quirks**
→ When viewing the newly generated project in an IDE file explorer (like VS Code or Cursor), if a directory contains only one sub-directory, the IDE often visually "squashes" them together on a single line (e.g., `3_crew/debate`). 
- **Workaround:** To force the IDE to expand the tree view and make navigation less confusing, you can create a temporary, empty directory alongside it (e.g., `mkdir other`).

**⭐ 🔹 CrewAI Default Project Structure**
→ The `crewai create crew` command generates a robust, standardized scaffolding for your project. Understanding this hierarchy is essential for navigating the framework.
1. **`knowledge/`**: Contains scaffolding files like `user_preference.txt`. This is where you would place background information or static documents that the models might need to reference.
2. **`src/[project_name]/`** (e.g., `src/debate/`): The core application directory. Because it uses a standard Python `src` layout, there are no other folders at this level.
3. **`src/[project_name]/config/`**: The most important folder for declarative logic. It contains two vital YAML files: `agents.yaml` and `tasks.yaml`. These come populated with default example configurations.
4. **`src/[project_name]/tools/`**: Contains `custom_tool.py`, which provides boilerplate code for building bespoke tools for your agents.
5. **`src/[project_name]/crew.py`**: The primary Python module that brings the team together. It utilizes Python decorators (`@Agent`, `@Task`, `@Crew`) to bind the YAML configurations to the Python execution logic.
6. **`src/[project_name]/main.py`**: The entry point script. This is the file you execute to run the crew (e.g., where `crew.kickoff()` is called).

> ⭐ **EXAM NOTE:** Understanding the default directory structure generated by `crewai create crew` is critical for building and maintaining CrewAI projects. You must know that agent/task definitions live in `src/[project_name]/config/` and the execution entry point is `main.py`.

**⭐ 🔹 Defining Agents Declaratively in `agents.yaml`**
→ CrewAI encourages defining agent characteristics in a YAML configuration file rather than hardcoding long text strings directly into Python. This separates prompt logic from execution logic.
- An agent definition requires specific key-value pairs:
  - **`role`**: The title or specific job function of the agent (e.g., "A compelling debater" or "Reporting Analyst").
  - **`goal`**: The specific objective the agent is trying to achieve during the run.
  - **`backstory`**: The persona, historical context, and behavioral guidelines for the agent.
- *Instructor Tip:* AI-assisted IDEs like Cursor are exceptionally good at auto-completing these natural language fields.

```yaml
# Example from the video
debater:
  role: >
    A compelling debater
  goal: >
    Present a clear argument either in favor of or against the motion. The motion is: {motion}
  backstory: >
    You're an experienced debater with a knack for giving concise but convincing arguments. The motion is: {motion}
```

> ⭐ **EXAM NOTE:** The declarative YAML configuration pattern is a foundational feature of CrewAI. You must understand how to map keys like `role`, `goal`, and `backstory` to configure an agent's behavior.

**⭐ 🔹 Template Variables in YAML Configuration**
→ You can inject dynamic data into your YAML prompts using template variables enclosed in curly braces, such as `{motion}` or `{topic}`.
- These variables act as placeholders when the application is built.
- The actual values are passed at runtime. When executing the crew in `main.py`, you provide an `inputs` dictionary to the kickoff function (e.g., `crew.kickoff(inputs={'motion': 'AI will replace software engineers'})`), which interpolates the value directly into the agent's goal and backstory.

> ⭐ **EXAM NOTE:** You must know how to pass dynamic data into agent prompts. It is done using `{variable_name}` syntax in the YAML file, which is then populated via the `inputs` dictionary passed into the crew's kickoff command in `main.py`.

**⭐ 🔹 The Technical Science Behind "Backstory"**
→ While a "backstory" sounds like creative roleplay, there is a strict scientific/technical reason it makes AI agents perform better.
- **The Concept:** Large Language Models (LLMs) fundamentally work by predicting the most likely next token based on the input sequence.
- **The Application:** When you inject a detailed backstory (e.g., "You are a fair judge with a reputation for weighing up arguments without factoring in your own views...") into the system prompt, you are providing massive framing context. 
- **The Result:** This context drastically alters the probability distribution of the model's output. By placing the model into a specific contextual "state," you increase the mathematical probability that the output tokens will be consistent with that persona, resulting in more accurate and stable agent behavior.

> ⭐ **EXAM NOTE:** Understanding *why* backstories work technically is a fundamental concept of prompt engineering for agents. They work by setting context in the prompt, which constraints and guides the LLM's next-token prediction probabilities to align with the desired persona.

**🔹 Overriding Default Models per Agent**
→ While the overall project has a default LLM set during the creation step, you can assign different, specific models to individual agents directly within the `agents.yaml` file.
- **Syntax:** Add the `model` or `llm` key to the agent's YAML definition, formatted as `[provider]/[model_name]`.
- **Example:** `model: openai/gpt-4o-mini`
- This allows for heterogeneous agent teams where a simple agent might use a cheap, fast model, while a complex reasoning agent uses a more expensive, advanced model.
