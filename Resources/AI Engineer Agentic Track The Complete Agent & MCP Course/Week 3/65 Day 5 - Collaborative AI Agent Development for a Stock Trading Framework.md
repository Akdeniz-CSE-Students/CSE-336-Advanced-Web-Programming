**🔹 Cleaning Up Boilerplate in `crew.py`**
→ The first step in defining the actual CrewAI structure involves navigating to the `crew.py` module. The instructor begins by deleting all the default, boilerplate code ("gunk") generated by the framework. The only structural component kept intact is the configuration hookup that links the Python class to the YAML configuration files (`agents.yaml` and `tasks.yaml`), as these are necessary to inject the predefined agent and task prompts into the execution environment.

**🔹 Defining the `engineering_lead` Agent**
→ The instructor uses the `@agent` decorator to define the first agent, the `engineering_lead`. While the AI coding assistant (Cursor) suggests enabling code execution for this agent, the instructor explicitly rejects this. The Engineering Lead's role is strictly to design the software architecture, not to run or test actual Python code. Therefore, `allow_code_execution=True` is omitted.
```python
@agent
def engineering_lead(self) -> Agent:
    return Agent(
        config=self.agents_config['engineering_lead'],
        verbose=True
    )
```

**⭐ 🔹 Backend Engineer & Safe Code Execution**
→ The second agent defined is the `backend_engineer`. Unlike the lead, this agent actually writes and runs code. To allow this, several critical parameters are introduced to the `Agent` configuration. Code execution is enabled, but it is strictly sandboxed inside a Docker container using `code_execution_mode="safe"`. The instructor also sets custom limits for how long the code can run and how many times the agent can attempt to fix errors.
- `allow_code_execution=True`: Gives the agent permission to run the code it generates.
- `code_execution_mode="safe"`: Ensures the code is executed securely inside a Docker container, preventing local system tampering.
- `max_execution_time=240`: Limits execution to a few minutes (4 minutes). The instructor notes the default/suggested 1000 seconds is too long.
- `max_retries=5`: Gives the agent up to 5 attempts to fix code if it fails to execute before the agent gets "upset" and errors out.

> ⭐ **EXAM NOTE:** Understanding how to safely enable code execution for agents is highly testable. You must know that `allow_code_execution=True` paired with `code_execution_mode="safe"` sandboxes the execution within a Docker container, and that guardrails like `max_execution_time` and `max_retries` are critical to prevent infinite loops or excessively long generation cycles.

```python
@agent
def backend_engineer(self) -> Agent:
    return Agent(
        config=self.agents_config['backend_engineer'],
        verbose=True,
        allow_code_execution=True,
        code_execution_mode="safe",
        max_execution_time=240,
        max_retries=5
    )
```

**⭐ 🔹 Frontend Engineer & UI Execution Constraints**
→ The `frontend_engineer` is defined next. The instructor makes a very deliberate architectural decision here: the frontend engineer is **not** given `allow_code_execution=True`. The reason is that frontend code in this stack often involves Gradio UIs. If the agent attempts to execute Gradio UI code inside a headless Docker container during the CrewAI run, it will attempt to spin up a web server/UI, which is a "different ballgame" and will break the containerized execution process. The frontend agent is restricted to only *writing* the code.

> ⭐ **EXAM NOTE:** This is a crucial architectural limitation to remember. UI code (like Gradio or Streamlit) should generally not be executed dynamically by agents inside sandboxed Docker environments during standard automated loops, as the process will hang or break trying to serve a web interface.

```python
@agent
def frontend_engineer(self) -> Agent:
    return Agent(
        config=self.agents_config['frontend_engineer'],
        verbose=True
    )
```

**⭐ 🔹 Crew Instantiation & The "Vibe Coding" LLM Overgeneration Warning**
→ When defining the `@crew`, the AI coding assistant attempts to explicitly list out every single agent and task (e.g., `agents=[self.engineering_lead(), self.backend_engineer()...]`). The instructor strongly advises against this and identifies it as a common issue with LLM "vibe coding." LLMs have a tendency to generate *too much* code and overcomplicate simple structures. Instead of hardcoding arrays, CrewAI provides `self.agents` and `self.tasks` properties automatically via the class decorators. The process is also set to execute sequentially.

> ⭐ **EXAM NOTE:** Recognizing LLM overgeneration (where the model attempts to explicitly hardcode variables instead of using built-in framework abstractions like `self.agents`) is a key troubleshooting skill. Utilizing the built-in decorators correctly makes the code more robust and less prone to manual updating errors.

```python
@crew
def crew(self) -> Crew:
    return Crew(
        agents=self.agents,
        tasks=self.tasks,
        process=Process.sequential,
        verbose=True
    )
```

**🔹 Setting up `main.py` and the `kickoff()` Function**
→ After finalizing the `crew.py` file, the instructor moves to `main.py` to trigger the execution. The standard boilerplate is deleted. A dictionary of inputs is defined based on the variables expected by the YAML files. Finally, the Crew is instantiated and started using the `.kickoff()` method, passing in the inputs.
```python
def run():
    inputs = {
        'requirements': requirements,
        'module_name': module_name,
        'class_name': class_name
    }
    try:
        EngineeringTeam().crew().kickoff(inputs=inputs)
    except Exception as e:
        raise Exception(f"An error occurred while running the crew: {e}")
```

**🔹 The Assignment: Trading Simulation Framework**
→ The instructor presents the prompt/input that will be passed into the Crew. The goal is to get the AI engineering team to build a complete "Account Management System for a Trading Simulation Platform."
The requirements passed to the AI include:
1. Allow users to create an account, deposit funds, and withdraw funds.
2. Record bought/sold shares and quantities.
3. Calculate the total value of the portfolio and profit/loss (PnL) from the initial deposit.
4. Report holdings at any point in time.
5. Report PnL at any point in time.
6. List all transaction history.
7. Implement business logic safeguards: prevent withdrawing into a negative balance, prevent buying shares the user cannot afford, and prevent selling shares the user doesn't own.
8. Integrate access to a mock `get_share_price(symbol)` function that returns fixed prices for AAPL, TSLA, and GOOGL.

**⭐ 🔹 Connection to the Final Project (MCP & Agent Traders)**
